# InkPi 评价依据与验证计划

## 1. 项目定位

InkPi 当前不是“自动给书法考级结果”的系统，而是一套面向：

- 楷书单字练习
- 行书单字练习
- 初学者与课堂反馈
- 设备端即时结果展示

的 **辅助评测系统**。

更稳妥的表述应该是：

- `total_score` 是统一主分
- 四维分是解释层，不直接替代教师终评
- 当前正式支持楷书与行书单字
- 其他书体与多字作品不在当前正式范围内

## 2. 四维评价依据

InkPi 的四维不是随意命名，而是把课堂上常见的书写观察维度映射到可计算特征。

| 维度 | 回答的问题 | 课堂观察点 | 当前对应特征 |
| --- | --- | --- | --- |
| 结构 | 字形重心、比例和空间分布是否稳定 | 重心、留白、主体位置、字框比例 | `center_quality`、`bbox_ratio`、`projection_balance`、`bbox_fill` |
| 笔画 | 笔画粗细、方向和控制是否稳定 | 粗细波动、转折、连带、纹理平顺性 | `texture_std`、`orientation_concentration`、`ink_ratio`、`component_norm` |
| 完整 | 主体是否完整、清晰、可识别 | 缺笔、断裂、粘连、噪声、边缘碰撞 | `ocr_confidence`、`dominant_share`、`edge_touch`、`subject_edge_safe` |
| 稳定 | 本次评分结果是否集中、可信、可复现 | 主分区间、概率集中度、重复评测波动 | `quality_confidence_norm`、`probability_margin_norm`、`feature_quality`、`score_range_fit` |

这里最重要的一点是：

- 四维用于解释“为什么是这个结果”
- 四维不反推主分
- 四维不宣称等同于正式考级标准

## 3. 双书体方法论

当前系统正式支持两种书体，但两者的解释口径不同：

### 3.1 楷书

楷书解释重点是：

- 结构规范
- 重心稳定
- 比例清楚
- 起收笔与边界清晰

因此在楷书中，系统更强调：

- `structure`
- `integrity`

### 3.2 行书

行书解释重点是：

- 连带关系
- 节奏感
- 流动性下的主体完整
- 在更动态书写中的识别稳定性

因此在行书中，系统更强调：

- `stroke`
- `stability`

这就是为什么当前架构改成“先 OCR，再按用户选择的 `script` 路由到对应 ONNX 模型”，而不是一套模型混评全部书体。

## 4. 当前支持边界

为了避免项目被误解为“万能书法 AI”，边界要说清楚：

### 4.1 正式支持

- 楷书单字
- 行书单字

### 4.2 当前不支持

- 隶书
- 草书
- 篆书
- 多字作品
- 自动书体识别

### 4.3 为什么这样定边界

这是参考公开研究和工程项目后的主动收缩：

- 规则性更强的书体更适合先做稳定评测
- 行书虽然更动态，但仍保留一定可识别结构
- 草书、篆书、多字作品对结构建模和上下文理解要求明显更高

因此，当前系统把“边界清楚、结果稳定”放在“支持范围看起来更大”之前。

## 5. 当前可量化的验证数据

系统当前已经能自动汇总下列量化指标：

- 总样本数
- 书体分布
- 覆盖汉字数
- 设备来源数
- 最近 7 天样本数
- 分数分布
- 四维均值
- 高频练习字
- 设备贡献度
- 专家复评覆盖率
- AI 与人工一致率
- AI 与人工平均分差

这些指标已经进入：

- Cloud API 汇总接口
- 小程序统计页
- reviewer-facing methodology payload

也就是说，项目后续答辩或汇报时，不必再手工拼凑样本数据。

## 6. 当前验证路线

为了回应“数据不够”和“评价依据不够权威”的问题，InkPi 当前采用以下验证路线：

1. 先验证主分与人工评分之间的总体一致性。
2. 再逐步校准四维解释层，而不是把四维当作现成标准答案。
3. 用设备来源、书体分布、时间分布和高频字信息分析数据偏差。
4. 为每条记录补充教师 / 专家复评入口，形成可回溯验证数据。

建议后续阶段目标：

- `500+` 份带标签样本
- `3+` 名教师或专家参与校核
- `20+` 名试用用户形成连续练习记录

## 7. 产品闭环

为了回应“不能只告诉用户写得好不好，还要给练习方向”的评审意见，当前版本已经补上：

- 小程序统计页：量化证据、验证目标、依据来源
- 小程序详情页：四维依据卡、阶段判断、优先提升项、练习建议
- 历史管理：单条删除、批量删除、筛选整理
- 云端复评接口：支持人工复评与评语写回
- 验证总览接口：直接返回复评覆盖率、一致率与平均分差

也就是说，项目已经从“出一个分数”逐步转向“给出结果、说明依据、记录过程、支持复核”的产品形态。

## 8. 与公开参考项目的关系

本文件配套的参考资料见：

- [参考项目与论文](./reference-projects-and-papers.md)

InkPi 在方法论上借鉴了公开项目和论文的三类思路：

- **系统型项目**：强调完整链路，而不是只写模型
- **评测型论文**：强调人工评分对照、数据集规模、评价指标
- **反馈型研究**：强调“score-only 不够”，需要给出分层反馈和练习建议

InkPi 的差异化点在于：

- 明确支持边界，只做楷书与行书单字
- 把树莓派小屏、运维后台、小程序和云同步放在同一套工程里
- 把评价依据与验证快照直接做进产品接口，而不是只写在论文里
