\begin{abstract}
针对书法学习中“反馈周期长、评价口径不稳定、设备端与云端脱节”的问题，本文设计并实现了一套面向楷书与行书单字练习的书法辅助评测系统 InkPi。系统以树莓派为设备端核心，采用 PyQt6 构建 3.5 寸小屏正式界面，在保持统一图像预处理与 OCR 入口不变的前提下，引入显式书体字段 \texttt{script}，将评测范围收敛为楷书与行书两种正式支持书体，并在 OCR 之后按书体路由到对应 ONNX 主评分模型。系统输出统一主分 \texttt{total\_score}，同时生成结构、笔画、完整、稳定四维解释分，用于教学反馈与结果说明。为支持真实部署，本文进一步实现了 Cloud API、运维后台、小程序和本地/云端数据库协同链路，使系统能够完成本地评测、结果同步、状态监控、移动端查看和复评数据沉淀。相比仅强调模型准确率的单模块原型，InkPi 更强调边缘设备可部署性、支持边界清晰性、解释层产品化以及验证数据的持续积累。本文还结合公开研究与项目，对系统定位、方法边界和验证路线进行了重构，使其更适合课堂辅助、答辩展示和后续产品化扩展。
\end{abstract}

\noindent\textbf{关键词：} 书法评测；树莓派；OCR；ONNX；可解释反馈；边缘云协同

\section*{Abstract}
InkPi is a Raspberry-Pi-centered calligraphy assistance system for single-character practice in regular script and running script. The system keeps image preprocessing and OCR shared across the whole runtime, while explicitly introducing a user-selected \texttt{script} field to route each sample to the corresponding ONNX scorer after OCR. The runtime produces a unified primary score together with four explanation dimensions, namely structure, stroke, integrity, and stability. Around this evaluation pipeline, the project further integrates a PyQt6 device-side interface for a 3.5-inch screen, a Flask-based Cloud API, a monitoring-oriented web operations console, and a miniapp for history browsing, statistics, and coaching-oriented result inspection. In contrast to score-only prototypes, InkPi emphasizes clear support boundaries, productized explanation outputs, cloud-synchronized evidence accumulation, and deployability on edge devices. This paper presents the system architecture, the dual-script evaluation workflow, related online projects and papers that informed the redesign, and the current validation strategy for deploying and evolving the system.

\noindent\textbf{Keywords:} calligraphy assessment; Raspberry Pi; OCR; ONNX; explainable feedback; edge-cloud system

\section{引言}
书法学习长期依赖教师面对面示范和点评，这种教学方式在专业训练中不可替代，但在初学者练习、课堂展示和设备端即时反馈场景中存在两个突出问题：其一，反馈周期长，学生难以在单次练习后立即获得结构性建议；其二，评价口径常停留在“写得好/不好”的经验描述，缺乏可记录、可传输、可复盘的数据出口。随着 OCR、边缘推理和移动端同步能力的成熟，书法辅助评测系统开始具备工程实现基础，但很多公开工作仍集中于单一模型或单一评测实验，较少把设备交互、部署、监控和移动端反馈纳入统一系统叙述。

InkPi 的目标并不是构建一个“自动给书法考级结论”的系统，而是构建一条更稳健的产品链路：树莓派设备端负责正式交互，统一预处理和 OCR 保持输入稳定；ONNX 模型负责输出统一主分；四维解释层负责将结果转化为更易理解的教学反馈；云端 API 负责结果沉淀与移动端分发；运维后台负责设备、模型和流程监控；小程序负责历史记录、统计指标和练习建议查看。换句话说，本项目优先回答的是“如何把评测能力组织成一个可部署、可解释、可持续验证的系统”，而不是单纯追求算法指标最大化。

在前期版本中，系统主要围绕楷书单字展开。随着评审意见逐步明确，我们将项目正式支持范围重构为“楷书 + 行书单字”，并显式去掉对其他书体的模糊承诺。这样做的原因有两点：一是楷书和行书在教学与练习场景中覆盖面更广；二是相比隶书、草书、篆书，多数结构与可识别性特征在这两类书体上更适合当前的工程路线。于是，InkPi 采用“用户手动选择书体，系统在 OCR 后按 \texttt{script} 路由评分模型”的策略，以保证支持边界清晰、实现成本可控、解释口径稳定。

从这些工作中，InkPi 形成了三条清晰判断。第一，书法评测不应只写成一个单模型任务，而应被组织为设备端、评分层、解释层、云端和移动端协同的系统。第二，评价问题天然带有主观性，因此“人工评分对照、样本量、验证指标、边界声明”必须进入正文，而不能只在答辩时口头补充。第三，仅给分数不足以支持学习改进，系统还需要提供面向用户的解释与建议。

相较于这些公开工作，InkPi 的差异化点在于：它把支持范围主动收敛为楷书与行书单字，把树莓派小屏、Cloud API、运维后台和小程序纳入统一仓库与统一部署链路，并将“评价依据”和“验证快照”做成了正式接口能力，而不是只存在于论文描述中。

\section{系统目标与支持边界}
当前版本的 InkPi 明确遵守以下边界：

\begin{itemize}
  \item 正式支持对象仅为楷书与行书单字；
  \item OCR 与预处理链路在两种书体之间共享；
  \item 新评测请求必须显式给出 \texttt{script}；
  \item \texttt{regular} 只路由到楷书模型，\texttt{running} 只路由到行书模型；
  \item 四维分用于解释，不作为正式考级结论；
  \item 隶书、草书、篆书以及多字作品不在当前正式范围内。
\end{itemize}

这种边界收缩并非能力退化，而是工程理性选择。CalliFormer 在讨论其结构先验时指出，规则性更强的楷书更适合依赖显式结构约束，而更流动的行书与草书会带来更大的建模难度 \cite{calliformer2025}。对于 InkPi 这样的边缘部署系统而言，只有在支持边界清晰时，解释逻辑、模型路由和验证口径才能同时保持稳定。

InkPi 当前系统由设备评测层、解释与存储层、云端协同层和运维监控层构成。整体流程如图~\ref{fig:system-flow} 所示。

  \centering
  \includegraphics[width=\textwidth]{figures/system-flow.png}
  \caption{InkPi 当前项目主链路与多端协同结构}
  \label{fig:system-flow}

从模块职责看，系统可以拆分为以下部分：

\begin{itemize}
  \item \textbf{Qt 设备端}：面向 480x320 小屏的正式交互入口，负责拍摄、上传、书体选择、结果展示和历史浏览；
  \item \textbf{预处理与 OCR}：统一执行图像质量校验、去网格、二值化、主体提取和单字识别；
  \item \textbf{双书体 ONNX 主评分}：在 OCR 之后按 \texttt{script} 路由至楷书或行书评分模型；
  \item \textbf{四维解释层}：基于质量特征、几何特征和主评分中间量生成解释分；
  \item \textbf{本地 SQLite}：保存结果、图像路径、四维分与调试字段；
  \item \textbf{Cloud API}：提供登录、结果上传、列表、详情、删除、方法论与验证相关接口；
  \item \textbf{运维后台}：监控主机、服务、双模型 readiness、硬件状态和后台输出；
  \item \textbf{小程序}：面向移动端查看历史、统计、建议、复评与验证快照。
\end{itemize}

与很多“设备端和 Web 端重复做同样的事”的系统不同，InkPi 在这一轮重构后明确区分了正式端与运维端的职责：Qt 只承担用户侧操作，Web 只承担监控与观察。这样既减轻了小屏交互负担，也使运维信息不再混入正式展示界面。

\section{双书体评测方法}
\subsection{共享预处理与 OCR}
图像预处理和 OCR 保持为项目共享入口。预处理阶段负责亮度与对比度校验、红色网格去除、二值化、主体提取和 OCR 裁剪。OCR 侧优先使用 PaddleOCR \cite{paddleocr}，在依赖不可用或设备资源不足时，通过 Cloud API 提供远程回退。InkPi 在这一层故意不做自动书体判定，因为书体判断错误会直接污染后续评分模型选择。

\subsection{显式书体选择与模型路由}
InkPi 采用显式书体输入策略。用户在 Qt 端或上传入口中明确选择 `regular` 或 `running`，系统在 OCR 之后执行如下路由：

\begin{equation}
\text{score\_model} =
\begin{cases}
f_{\text{regular}}, & \text{if } script = \texttt{regular} \\
f_{\text{running}}, & \text{if } script = \texttt{running}
\end{cases}
\end{equation}

其中，$f_{\text{regular}}$ 和 $f_{\text{running}}$ 分别对应楷书与行书的 ONNX 主评分模型。若请求未显式提供书体，或目标模型文件不存在，系统会直接拒绝评测并返回明确错误，而不是隐式退回到某个默认模型。这样做的意义在于：系统不假装“什么都支持”，而是在支持范围内保证行为确定。

\subsection{主分与四维解释层}
InkPi 保留统一主分 \texttt{total\_score} 作为唯一主口径，同时输出四维解释分：

\begin{itemize}
  \item \textbf{结构（structure）}：字形重心、比例与空间分布是否稳定；
  \item \textbf{笔画（stroke）}：笔画粗细、方向、转折与连带控制是否稳定；
  \item \textbf{完整（integrity）}：主体是否完整、清晰、可识别；
  \item \textbf{稳定（stability）}：当前结果是否集中、可信、可复现。
\end{itemize}

四维层复用主评分中的质量特征和校准中间量，但解释文本按书体分别生成。楷书更强调结构规范、重心和边界；行书则更强调流动性下的结构完整与识别稳定。因此，四维名称相同，但维度说明、最弱项建议和练习提示会随着 \texttt{script} 改变。

\subsection{评价依据与验证快照}
为了回应“评价标准依据不足”的问题，InkPi 在结果结构中引入了方法论载荷，包括：

\begin{itemize}
  \item 支持边界说明；
  \item 书体对应的练习画像；
  \item 四维依据卡；
  \item 验证快照，如样本量、覆盖汉字数、设备来源数、复评覆盖率和目标进度。
\end{itemize}

这意味着“为什么这样评”和“当前验证做到哪一步”都已经进入产品接口，不再只是论文或答辩话术。

\subsection{运维后台}
Web 端已经从早期的“重复评测页”重构为运维后台。前端使用 React 构建，后端通过 Flask 暴露状态快照与日志接口。当前后台重点监控：

\begin{itemize}
  \item 主机状态：CPU、内存、磁盘、网络与温度；
  \item 服务状态：Qt、WebUI、Cloud API；
  \item 模型状态：本地 OCR、远程 OCR、楷书模型、行书模型、四维解释层；
  \item 硬件状态：摄像头、GPIO、LED、I2C、SPI、音频；
  \item 流程状态：最近评测结果、书体分布、日志输出。
\end{itemize}

这使得 Web 端真正承担起“设备运行中心”的角色，而不是继续与 Qt 重复交互功能。

\subsection{云端与小程序}
Cloud API 负责登录、结果上传、历史列表、详情查询、删除、方法论读取和验证总览。小程序则围绕云端数据完成三类工作：

\begin{itemize}
  \item 历史记录与详情查看；
  \item 按书体查看分布、趋势、四维均值和高频练习字；
  \item 查看依据卡、优先提升项与练习建议。
\end{itemize}

相比只展示一张成绩卡，这一设计使小程序能够承担“学习闭环”中的记录、对比和跟练建议功能。

\subsection{训练与部署分离}
当前训练策略明确为：模型训练在本机 V100 工作站上完成，树莓派与公网后端只消费导出的 ONNX 文件。这种设计来自 ONNX Runtime 的部署优势 \cite{onnxruntime}，也符合边缘设备算力受限的现实约束。GitHub Actions 只负责校验双模型产物是否存在、接口结构是否一致，而不在默认 CI 中直接训练模型。

\section{验证与讨论}
\subsection{当前验证方式}
InkPi 当前的验证并不把重点放在“单个模型指标的极限值”上，而是放在“完整链路是否可运行、边界是否清晰、结果是否可追踪”上。当前版本已完成以下类型验证：

\begin{itemize}
  \item Python 单元测试：本地数据库、Cloud API、WebUI 和云同步链路；
  \item 小程序脚本检查：历史页、详情页、统计页；
  \item 论文工程编译：XeLaTeX 成功输出 PDF；
  \item 树莓派实机联调：Qt 全屏、Cloud API、运维后台和 480x320 布局检查。
\end{itemize}

\subsection{量化验证的当前状态}
评审意见表明，项目不能只说“以后会继续收集数据”，而必须把可量化验证纳入当前系统。因此，InkPi 已将以下指标写入正式统计链路：

\begin{itemize}
  \item 总样本数与按书体样本分布；
  \item 覆盖汉字数；
  \item 设备来源数；
  \item 最近时段样本增长趋势；
  \item 分数分布与四维均值；
  \item 专家复评覆盖率、一致率与平均分差。
\end{itemize}

这些指标的意义不在于当前已经拥有大规模权威数据，而在于系统已经具备持续沉淀证据和对外展示验证状态的能力。

\subsection{局限性}
InkPi 当前仍有明显局限：

\begin{itemize}
  \item 正式支持范围只覆盖楷书与行书单字；
  \item 四维解释层仍属于工程化解释，不等同于标准化考级体系；
  \item 真实专家复评规模仍需继续扩大；
  \item 树莓派端的摄像头、GPIO 与外设能力仍依赖具体硬件环境。
\end{itemize}

不过，这些局限已经通过支持边界、方法论说明与验证快照被显式暴露出来，而不是被隐去。

\section{结论}
本文围绕 InkPi 项目给出了一套完整的系统设计与实现方案。相较于单纯强调模型的书法评测原型，InkPi 更关注从边缘设备到云端再到移动端的完整工程闭环。系统在保持统一 OCR 与预处理入口不变的前提下，将正式支持范围重构为楷书与行书单字，并通过显式书体选择和双模型路由保证了行为边界清晰。与此同时，项目通过四维解释层、方法论载荷、运维后台和小程序统计链路，把“分数、依据、建议、验证状态”组合成一个更适合教学与展示的产品形态。

后续工作可以从三个方向继续展开：第一，继续扩大双书体样本和专家复评规模，提升评价可信度；第二，引入更规范的 HTTPS、域名与账户体系，提升公网部署成熟度；第三，针对树莓派场景进一步增强离线容错、硬件探测和课堂使用体验。总体来看，InkPi 已经不再只是一个“能打分”的演示系统，而是一套边界清晰、可解释、可部署、可继续积累证据的书法辅助评测平台。
