\begin{abstract}
针对书法练习过程中“反馈滞后、评价主观、设备形态分散”的问题，本文设计并实现了一套面向单字书法场景的智能评测系统 InkPi。系统以树莓派为设备端核心，采用 PyQt6 构建 3.5 寸小屏交互界面，使用图像预处理、OCR 识别、ONNX 质量评分与四维解释评分构成主评测链路，并通过 Cloud API、运维后台与微信小程序实现结果同步、运行监控与移动端查看。项目在工程上强调“产品化端侧界面 + 可解释评分 + 云端协同 + 实时运维”四个方向：主分保留为稳定的总分输出，同时引入“结构、笔画、完整、稳定”四维解释层用于教学反馈；Web 端不再承担重复评测操作，而转为设备运行中心，用于监控模型加载、评测流程、硬件健康与后台日志。本文重点介绍系统架构、评测方法、关键实现与部署验证过程。结果表明，该系统已经能够完成单字图像的本地评测、结果存储、云端同步以及多端展示，并具备进一步扩展为实际教学产品的基础。
此外，针对“评价依据不足”和“样本量化不足”的评审意见，当前版本已经把四维评价依据、量化证据快照、验证目标与练习建议整合到云端接口和小程序中，使系统不仅能给出结果，还能说明“为什么这样评”“当前数据到哪一步”“下一步该怎么练”。
\end{abstract}

\noindent\textbf{关键词：} 书法评测；树莓派；OCR；ONNX；可解释评分；多端协同

\section*{Abstract}
InkPi is a dual-script single-character calligraphy evaluation system that integrates Raspberry Pi edge hardware, a PyQt6 small-screen interface, OCR recognition, script-specific ONNX quality scoring, four-dimension explanation scoring, cloud synchronization, and an operations console. The current release formally supports only two scripts: regular script and running script. OCR and preprocessing remain shared across the whole runtime, while the user manually selects the script and the system routes the sample to the corresponding ONNX scorer after OCR. The paper presents the architecture, evaluation pipeline, explanation layer, UI implementation, deployment workflow, and the dual-script rebuild of the current system. The implemented stack supports local evaluation, cloud result synchronization, a mobile miniapp with quantitative statistics and record management, and a monitoring-oriented web console that tracks host status, model readiness, hardware availability, and runtime logs. The project also includes a GitHub Actions based CI/CD chain with four dedicated tracks: pull-request regression validation, public backend deployment, Raspberry Pi full-stack deployment, and miniapp release-candidate packaging. The latest version exposes evaluation-basis cards, validation snapshots, expert-review endpoints, agreement-rate statistics, score-gap summaries, and coaching-oriented follow-up suggestions so that the system can respond to reviewer concerns about authority, evidence, and product completeness. Model training and export are performed on a local V100 workstation, while Raspberry Pi devices and public backend nodes only consume exported ONNX artifacts for inference.

\noindent\textbf{Keywords:} calligraphy evaluation; Raspberry Pi; OCR; ONNX; interpretable scoring; edge-cloud collaboration

\section{引言}
书法练习具有明显的视觉反馈依赖与教师经验依赖。传统教学通常依赖教师面对面点评，反馈周期长且难以量化；而通用 OCR 或图像分类系统又难以直接回答“这个字写得怎么样”和“问题主要出在哪里”这两个教学场景中的核心问题。因此，单字书法智能评测系统需要同时满足以下要求：一是能在边缘设备上完成拍摄、评测与结果展示；二是输出稳定的主分与可理解的解释分；三是支持云端同步与移动端查看；四是具备运维与状态可视化能力，方便在真实设备环境中调试和维护。

基于上述需求，本文围绕当前项目 InkPi 的工程实现展开。该项目不是单纯的算法验证脚本，而是一套由设备端、后端与移动端共同组成的完整系统。设备端使用树莓派与小尺寸触摸屏作为正式交互入口，运维后台则承担运行状态监控与后台输出观察，云端接口负责结果汇聚与远程 OCR 回退，小程序用于移动端查看历史与详情。项目的目标是将书法图像评测从“单模块演示”推进为“能够部署、能够维护、能够解释”的完整产品链路。

\section{系统总体架构}
InkPi 当前系统由四个主要层次构成：设备评测层、解释评分层、云端协同层与运维监控层。设备评测层负责采集或上传书法单字图像，并在本地完成预处理、OCR 与 ONNX 评分；解释评分层负责从主评分链路中提取可解释特征，生成四维解释分；云端协同层负责结果同步、历史查询与移动端展示；运维监控层则通过 Web 运维后台实时观察设备、模型与服务状态。

\section{单字评测方法设计}
\subsection{图像预处理与前置质量校验}
为了提高 OCR 与评分模型的稳定性，InkPi 在图像进入识别链路前执行统一的预处理与前置校验。该阶段主要包括五类操作：

\begin{itemize}
  \item 亮度、对比度与内容有效性预检，用于快速拦截过暗、过亮、对比度不足或明显无主体的输入；
  \item 可选透视矫正，用于处理拍摄角度偏斜导致的纸面变形；
  \item 缩放与灰度化，将不同来源图像转换到统一分析尺度；
  \item HSV 红色网格去除，用于滤除米字格等红色辅助背景；
  \item 自适应二值化、去噪与主体提取，为 OCR 与质量评分生成更稳定的前景区域。
\end{itemize}

其中，前置质量校验并不仅仅判断“能不能识别”，还会检测墨迹比例、连通域分布、边缘复杂度和主体集中度，以避免杂乱背景、遮挡或非单字内容误入主链路。该策略的意义在于：如果在预处理前就发现画面不具备基本可评测性，就应尽早给出用户提示，而不是让后续 OCR 和评分模型输出低质量或不可解释的结果。

\subsection{OCR 识别与云端回退}
InkPi 使用 PaddleOCR 作为本地 OCR 主体\cite{paddleocr}。系统优先尝试在设备或公网后端本地运行 OCR；若设备端无法加载本地 OCR，或运行环境为资源较弱的平台，则可以通过 \texttt{INKPI\_CLOUD\_BACKEND\_URL} 与设备密钥将 OCR 请求回退到 Cloud API。这样设计有两个优势：

\begin{itemize}
  \item 在 x86\_64 环境下可直接利用本地 PaddleOCR，提高独立运行能力；
  \item 在 ARM 或依赖不齐全的场景中仍能通过远端 OCR 保持链路完整。
\end{itemize}

OCR 输出不仅包括识别字符本身，还包括识别置信度与候选框信息。其中置信度被后续评分与解释层重复利用，用于衡量当前样本的“可识别性”和“完成稳定度”。

\subsection{主评分模型}
主评分链路使用 ONNX Runtime 加载质量评分模型\cite{onnxruntime}。模型输入包括预处理后的 ROI、字符编码与 OCR 置信度，输出三档质量概率 \texttt{bad / medium / good}，并进一步融合为 \texttt{total\_score}、\texttt{quality\_level} 和 \texttt{quality\_confidence}。项目中保留主分作为唯一的官方评分出口，其作用是保证不同端侧界面、历史记录与云端同步都围绕同一套主分口径展开。

在实现上，主评分服务并不只依赖模型原始 logits，还结合了一组质量特征进行分段校准。这些特征包括前景比例、包围盒比例、中心质量、连通域归一化、边缘接触率和纹理标准差。通过引入校准层，系统可以在模型输出和工程稳定性之间取得平衡，避免主分被单一概率值直接决定。

\subsection{四维解释评分}
为了回答“为什么是这个分数”这一问题，InkPi 在主分之外新增了四维解释层。四维分并不反推主分，而是服务于教学反馈、界面展示与调试分析。四个维度定义如下：

\begin{itemize}
  \item \textbf{结构}：反映重心、布局和主体在画面中的组织稳定性；
  \item \textbf{笔画}：反映笔画粗细、方向与局部纹理是否平稳；
  \item \textbf{完整}：反映主体是否清晰、完整、易于识别；
  \item \textbf{稳定}：反映主模型置信度、概率间隔与特征信号的一致性。
\end{itemize}

四维解释分采用归一化特征的线性组合。若将所有参与项归一化到 $[0,1]$，则当前实现可概括为：

\begin{align}
S_{\text{structure}} &= 100 \times (0.35 c_{\text{center}} + 0.25 b_{\text{ratio}} + 0.20 p_{\text{balance}} + 0.20 b_{\text{fill}}), \\
S_{\text{stroke}} &= 100 \times (0.30 t_{\text{texture}} + 0.25 o_{\text{orientation}} + 0.25 i_{\text{ink}} + 0.20 c_{\text{component}}), \\
S_{\text{integrity}} &= 100 \times (0.35 o_{\text{ocr}} + 0.25 d_{\text{dominant}} + 0.20 (1-e_{\text{edge}}) + 0.20 s_{\text{subject}}), \\
S_{\text{stability}} &= 100 \times (0.40 q_{\text{conf}} + 0.25 p_{\text{margin}} + 0.20 f_{\text{quality}} + 0.15 r_{\text{fit}}).
\end{align}

式中，$c_{\text{center}}$ 表示中心质量，$b_{\text{ratio}}$ 表示包围盒带状得分，$p_{\text{balance}}$ 表示投影平衡度，$b_{\text{fill}}$ 表示主体填充度；$t_{\text{texture}}$、$o_{\text{orientation}}$、$i_{\text{ink}}$ 和 $c_{\text{component}}$ 分别描述纹理、方向、墨迹比例与连通域质量；$o_{\text{ocr}}$ 与 $q_{\text{conf}}$ 分别来自 OCR 置信度和主模型概率；$p_{\text{margin}}$、$f_{\text{quality}}$ 和 $r_{\text{fit}}$ 则来自主评分校准过程中的中间量。

这种设计的关键在于：主分维持统一口径，解释层提供教学说明。实际界面中，Qt 和小程序只展示四维分及其“最强项 / 待提升项”，而 Web 运维后台和本地详情接口则保留 \texttt{score\_debug}，用于查看概率、几何特征与校准中间值。

\subsection{评价依据与验证边界}
为了避免四维解释分被误解为“AI 自行定义的权威等级”，InkPi 在当前版本中明确加入了评价依据层。该层并不改变主分计算，而是对每个维度补充三个信息：其一，维度到底在观察什么书写问题；其二，该维度当前对应哪些可计算特征；其三，用户可以如何围绕该维度继续练习。换句话说，系统不再只输出“结构 82 分”，而是同步给出“结构关注重心、比例和空间分布”“该分值主要来自 \texttt{center\_quality}、\texttt{bbox\_ratio}、\texttt{projection\_balance} 等特征”“下一步建议先用米字格检查重心”等解释信息。

与此同时，系统也明确给出当前项目边界：第一，主分 \texttt{total\_score} 仍是唯一统一口径；第二，四维分属于解释层，不直接替代教师终评；第三，当前阶段正式支持楷书单字与行书单字，并继续面向初学者训练；第四，隶书、草书、篆书及多字作品当前不在正式支持范围内。这样的边界声明有助于把项目定位在“辅助评测系统”而不是“自动考级系统”，从而使方法论表述更稳健。

\section{系统实现}
\subsection{设备端正式界面}
设备端使用 PyQt6 构建正式交互界面，重点适配树莓派 3.5 寸 $480 \times 320$ 横屏。界面采用深色背景与 Windows 10 深蓝风格的产品化设计，强调大字号、清晰层级与触控友好布局。首页突出“开始评测”，拍摄页聚焦取景与上传，结果页则采用“主分大卡片 + 四维评分卡 + 历史入口”的结构，以保证在小尺寸屏幕上仍具备良好的可读性和操作性。针对 2026-04-19 的真机联调结果，当前版本进一步压缩了拍摄页头部控件、结果页指标区与底部操作区的垂直占用，避免主按钮、四维分值和关键反馈在真实 3.5 寸屏幕上发生裁切。

\subsection{运维后台与实时监控}
随着系统从“单端评测”扩展为“设备 + 云端 + 小程序”的完整工程，仅靠结果页已经无法满足维护需求。因此，项目将 WebUI 改造为运维后台：前端采用 React 编写，并引入 Three.js 背景氛围层；后端则通过 Flask 对外暴露主机状态、模型加载、硬件健康、服务栈状态、评测流水与日志缓冲接口。运维后台通过 SSE 持续拉取快照更新，其主要监控内容包括：

\begin{itemize}
  \item 主机层：温度、内存、磁盘、负载、IP 地址；
  \item 服务层：Cloud API、Web 运维后台、Qt 正式端的运行与健康状态；
  \item 模型层：本地 OCR、远端 OCR、ONNX 评分模型与四维解释层的就绪情况；
  \item 硬件层：摄像头、音频、LED、GPIO、SPI 与 I2C 的可用性；
  \item 流程层：图像进入、预处理、OCR、主评分、解释评分与存储阶段的事件记录；
  \item 输出层：应用日志缓冲与运行日志尾部内容。
\end{itemize}

这种设计使 Web 端从“重复操作页面”转变为“真实运维中心”，更适合面向多设备部署和后期维护。

\subsection{云端接口与移动端展示}
Cloud API 采用 Flask 构建，提供登录、结果上传、结果列表、结果详情、设备 OCR 与批量删除等接口。设备端完成本地评测后，会将结果写入本地 SQLite，并通过设备密钥上传到云端。云端数据库统一存储主分、等级、四维解释分与详情字段，从而使小程序可以只承担结果查看，不必重复执行评测。

在当前项目中，小程序承担两类任务：一是按时间查看历史评测结果，二是在详情页展示总分、等级、四维分与反馈信息。相比设备端，小程序更强调轻量浏览与异步访问；相比运维后台，小程序不展示调试字段与底层日志。

在本轮改进中，小程序不再只是“看结果”的轻量壳层，而是开始承担产品化闭环：历史页支持筛选、单条删除与批量删除；统计页新增样本量、覆盖汉字数、设备来源数、分数分布、四维均值、评价依据来源与验证目标；详情页新增四维依据卡、阶段判断、优先提升项与练习动作建议。这样一来，移动端既能给用户看成绩，也能回答评审最关心的三个问题：当前数据有多少、评价依据是什么、系统准备如何继续验证。

\section{部署与验证}
InkPi 当前已经具备跨设备部署能力。设备侧采用树莓派 + 小屏完成正式评测与展示；公网后端部署在 Debian 12 服务器，用于承载 Cloud API 与运维后台；移动端通过小程序连接云端接口完成结果读取。部署后，项目的验证工作主要围绕三个方面展开。

\subsection{功能回归验证}
项目当前保留了多组回归测试脚本，包括 Web 接口、Cloud API、云同步和综合链路验证。典型检查项包括：

\begin{itemize}
  \item \texttt{test\_web\_ui.py}：验证 Web 后端接口与结果读取；
  \item \texttt{test\_all.py}：验证评测核心服务与数据结构；
  \item \texttt{test\_cloud\_api.py}：验证云端接口认证、列表与详情；
  \item \texttt{test\_cloud\_ocr\_api.py}：验证设备 OCR 接口；
  \item \texttt{test\_cloud\_sync\_integration.py}：验证本地结果同步到云端的完整流程。
\end{itemize}

\subsection{量化证据与验证计划}
除了功能是否打通之外，InkPi 当前还把“可量化证据”直接做成系统输出。云端汇总接口会自动返回总样本数、覆盖汉字数、设备来源数、最近 7 天样本数、分数分布、四维均值、高频练习字和设备贡献度；小程序统计页则把这些指标与验证目标放在同一页面中展示。这样做的价值在于，项目在答辩或汇报时不需要再手工整理零散数据，而可以直接给出“目前有多少样本、离目标还差多少、主要样本来自哪些设备”的量化说明。

从验证策略上看，当前版本并没有把四维分包装成官方等级标准，而是保留了“人工对照优先”的原则。系统规划的下一阶段目标包括：积累不少于 500 份带标签样本，引入不少于 3 名教师或专家参与校核，并组织不少于 20 名试用用户形成连续练习记录。该计划的核心思想是先验证主分与人工判断的一致性，再逐步校准四维解释层，从而让项目从“可演示”迈向“可验证”。

\section{结论与展望}
本文围绕 InkPi 当前项目给出了一套完整的系统设计与实现方案。与只验证模型效果的原型不同，InkPi 已经形成由设备端、解释评分层、云端接口、运维后台与移动端组成的完整工程链路。系统在方法上保留了稳定的主分，同时通过四维解释层增强可读性；在实现上完成了小屏设备端产品化、Web 运维后台转型、云端同步与公网部署；在工程上具备继续扩展为真实教学产品的基础。

同时，项目已经开始把“评价依据”“量化证据”和“练习建议”纳入正式产品流程，这使得 InkPi 不再只是输出分数，而是逐步形成一套能够面向评审、教师和学习者同时解释的系统表达。

后续工作可以从三个方向展开：第一，继续补充真实书法样本，完善主评分模型与四维解释分的标注体系；第二，引入 HTTPS、域名与更完善的账号体系，进一步提升公网服务的可用性；第三，在树莓派端增加更稳定的硬件检测、自启动与离线容错能力，使系统更适合长期部署在课堂或展示场景中。
